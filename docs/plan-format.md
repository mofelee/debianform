# DebianForm Plan Format

<p align="right"><strong>English</strong> | <a href="plan-format.zh.md">简体中文</a></p>

This document defines the current alpha format emitted by `dbf plan --format json`.
It is also the input model for the terminal tree diff and the static preview
generated by `dbf plan --html plan.html`. See the
[compatibility policy](compatibility-policy.md) for format-version compatibility,
breaking-change rules, and guidance for consumers.

The goals of the plan format are to:

- Represent user-facing addresses consistently.
- Preserve structured, field-level diffs.
- Support line-level diffs for text content.
- Never disclose sensitive plaintext.
- Associate low-level provider resources in debug mode.
- Let CI systems, auditing systems, and HTML/TUI viewers reuse the same data.

## Top-Level Structure

```json
{
  "format_version": "debianform.plan.alpha1",
  "generated_at": "2026-06-20T12:00:00Z",
  "command": {
    "file": "examples/fleet.dbf.hcl",
    "host": "server1"
  },
  "summary": {
    "create": 1,
    "update": 2,
    "delete": 0,
    "no_op": 0,
    "operations": 2
  },
  "changes": [],
  "operations": [],
  "diagnostics": []
}
```

Field requirements:

- `format_version` must change whenever the format changes incompatibly.
- `generated_at` uses UTC RFC3339.
- `changes` contains changes to resources or domain objects.
- `operations` contains action nodes triggered by changes, such as an APT cache
  refresh or nftables activation.
- `diagnostics` contains warnings or non-fatal information; a fatal error must
  not produce a successful plan.

## Action Enumeration

```text
create
update
delete
destroy
no_op
replace
read
run
```

Meanings:

- `delete` means the object is still declared in configuration but is expected
  to be absent.
- `destroy` means the object was removed from configuration and must be
  destroyed according to its ownership.
- `replace` means an in-place update is impossible, so delete/create or an
  atomic replacement is required.
- `run` is used for operation nodes.

## PlanChange

```json
{
  "host": "server1",
  "address": "host.server1.nftables.file[\"20-services\"]",
  "action": "update",
  "summary": "update nftables snippet 20-services",
  "source": {
    "file": "examples/fleet.dbf.hcl",
    "line": 560,
    "path": "host.server1.nftables.file[\"20-services\"]"
  },
  "provider_address": "nftables_file.server1_20_services",
  "delete_behavior": "external-side-effect",
  "delete_notes": [
    "removes nftables file",
    "may trigger nftables validate or activate operations"
  ],
  "delete_risk": "high",
  "diff": {},
  "low_level_actions": []
}
```

Requirements:

- `address` is a stable address that users can understand.
- `host` is the explicit logical host that owns the resource; it does not need
  to be parsed from `address`.
- `provider_address` is emitted only by `dbf plan --debug`; normal JSON,
  terminal, and HTML output omit it.
- `source` points to the corresponding location in the user's DSL.
- `delete_behavior`, `delete_notes`, and `delete_risk` are emitted only for
  `delete`, `destroy`, and `forget`-style actions.
- `diff` uses a `DiffNode`.
- `low_level_actions` explains the underlying actions that will actually run,
  but does not replace operation nodes.

Deletion behavior fields:

- `delete_behavior` is a machine-readable classification such as `forget`,
  `remove-managed-artifact`, `restore-original`, `destructive`,
  `external-side-effect`, or `unknown`.
- `delete_notes` is an array of short explanations. It contains neither ANSI
  color codes nor sensitive plaintext.
- `delete_risk` is a risk level such as `low`, `medium`, `high`, or `unknown`.

## DiffNode

```json
{
  "path": ["content"],
  "kind": "text",
  "action": "update",
  "sensitive": false,
  "before": null,
  "after": null,
  "children": [],
  "hunks": []
}
```

`kind` enumeration:

```text
object
map
set
list
scalar
text
sensitive
```

Rules:

- The root node uses `kind = "object"`; field changes appear in its
  deterministically sorted `children`.
- Objects and maps are diffed recursively by key.
- Sets are diffed by element identity.
- Lists are diffed by index only when order is meaningful.
- Labeled block lists must first be normalized into maps.
- Scalar diffs use `before` and `after`.
- Text diffs use `hunks`.
- Sensitive diffs must not emit plaintext.
- File or systemd unit content derived from a sensitive component input must
  also use a sensitive diff that emits only SHA-256 and byte-count summaries.
- The terminal and HTML renderers must consume the same `DiffNode` tree; they
  must not calculate differences independently.

## Text Diff

```json
{
  "path": ["content"],
  "kind": "text",
  "action": "update",
  "hunks": [
    {
      "old_start": 8,
      "old_lines": 1,
      "new_start": 8,
      "new_lines": 1,
      "lines": [
        {
          "op": "delete",
          "text": "add rule inet filter input tcp dport { 22, 80 } accept"
        },
        {
          "op": "create",
          "text": "add rule inet filter input tcp dport { 22, 80, 443 } accept"
        }
      ]
    }
  ]
}
```

`op` uses `create`, `delete`, and `no_op` from the same action enumeration.

## Sensitive Diff

```json
{
  "path": ["content"],
  "kind": "sensitive",
  "action": "update",
  "sensitive": true,
  "before_summary": {
    "sha256": "old-redacted-sha256",
    "bytes": 64
  },
  "after_summary": {
    "sha256": "new-redacted-sha256",
    "bytes": 64
  }
}
```

Requirements:

- `before_summary` and `after_summary` should come from the resource's
  `summary`; they may also come from `content_sha256` and `content_bytes`.
- Neither the JSON nor the text renderer may contain sensitive plaintext.
- Secret plaintext is forbidden.
- `sha256` can be used to identify a change, but is not secure encrypted
  storage.
- If the hash itself could present a disclosure risk, the provider may emit
  only `changed: true` and a size range.

## OperationNode

```json
{
  "host": "server1",
  "address": "host.server1.nftables.activate",
  "action": "run",
  "summary": "activate nftables ruleset",
  "depends_on": [
    "host.server1.nftables.validate"
  ],
  "triggered_by": [
    "host.server1.nftables.file[\"20-services\"]"
  ],
  "command_preview": "nft -f /etc/nftables.conf",
  "source": {
    "file": "examples/fleet.dbf.hcl",
    "line": 560,
    "path": "host.server1.nftables.file[\"20-services\"]"
  }
}
```

An `OperationNode` describes a meaningful execution action, not an arbitrary
shell hook. Typical addresses include:

`host` is the operation's explicit execution target. The scheduler, provider,
and HTML filter all use this field; the address is not responsible for routing.

```text
host.server1.apt.cache_refresh
host.server1.nftables.validate
host.server1.nftables.activate
host.server1.systemd.daemon_reload
host.server1.services.service["myapp"].restart
```

## Diagnostics

```json
{
  "severity": "warning",
  "message": "The WireGuard private key source is local; keep examples/secrets out of git.",
  "source": {
    "file": "examples/fleet.dbf.hcl",
    "line": 410,
    "path": "host.server1.secrets.file[\"/etc/wireguard/private.key\"]"
  }
}
```

The first version supports these `severity` values:

```text
info
warning
```

An error-level diagnostic must make the command fail instead of producing an
applicable plan.
