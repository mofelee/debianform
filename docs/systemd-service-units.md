# systemd Service Units

<p align="right"><strong>English</strong> | <a href="systemd-service-units.zh.md">简体中文</a></p>

DebianForm supports two ways to define a `.service` unit:

- `systemd.unit`: write the complete systemd unit file as plain text.
- `systemd.service_unit`: describe a typical service structurally and let
  DebianForm generate the `.service` unit file.

Both forms ultimately compile into the same `SystemdUnit`, are written to
`/etc/systemd/system/*.service`, and trigger `systemctl daemon-reload` when the
content changes. Whether the service starts at boot and whether it is currently
running are still managed by `services.service`.

## Plain-Text Form

The plain-text form is appropriate when you need full control of unit content,
such as complex `ExecStartPre=` directives, `CapabilityBoundingSet=`, directives
not yet abstracted across multiple drop-ins, or a non-service unit type.

```hcl
systemd {
  unit "myapp.service" {
    content = <<-EOF
      [Unit]
      Description=My App
      Wants=network-online.target
      After=network-online.target

      [Service]
      WorkingDirectory=/var/lib/myapp
      Environment=MYAPP_ENV=production
      ExecStart=/usr/local/bin/myapp --config /etc/myapp/config.yaml
      Restart=always
      RestartSec=5s

      [Install]
      WantedBy=multi-user.target
    EOF
  }
}

services {
  service "myapp" {
    enabled = true
    state   = "running"
  }
}
```

If you only need to write a `.service` as plain text, you can also use the text
mode of `service_unit`; `.service` is appended to the label automatically:

```hcl
systemd {
  service_unit "myapp" {
    content = <<-EOF
      [Service]
      ExecStart=/usr/local/bin/myapp --config /etc/myapp/config.yaml
    EOF
  }
}
```

## Structured Form

The structured form is suitable for typical long-running services. It reduces
repeated boilerplate while preserving the clear division of responsibility with
`services.service`: `service_unit` generates only the unit file, while
`services.service` manages only its enabled and running state.

```hcl
systemd {
  service_unit "myapp" {
    description = "My App"

    run = [
      "/usr/local/bin/myapp",
      "--config",
      "/etc/myapp/config.yaml",
    ]

    working_dir   = "/var/lib/myapp"
    restart       = "always"
    restart_delay = "5s"

    wants = ["network-online.target"]
    after = ["network-online.target"]

    environment = {
      MYAPP_ENV = "production"
    }

    stdout = "journal"
    stderr = "journal"
  }
}

services {
  service "myapp" {
    enabled = true
    state   = "running"
  }
}
```

The structured form above generates the equivalent
`/etc/systemd/system/myapp.service`:

```ini
[Unit]
Description=My App
Wants=network-online.target
After=network-online.target

[Service]
WorkingDirectory=/var/lib/myapp
Environment=MYAPP_ENV=production
ExecStart=/usr/local/bin/myapp --config /etc/myapp/config.yaml
Restart=always
RestartSec=5s
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

## Comparison

| Capability | Plain-text `systemd.unit` | Structured `systemd.service_unit` |
| --- | --- | --- |
| Unit type | Any unit filename | `.service`; the label may omit `.service` |
| Content control | Entirely handwritten | Generated by DebianForm from fields |
| Common service boilerplate | Must be written manually | Built-in `run`, environment, working directory, restart, logging, and dependency fields |
| Uncommon systemd directives | Write them directly | Use plain text when they are not yet covered |
| File metadata | `owner`, `group`, `mode` | `owner`, `file_group`, `mode` |
| Service runtime state | Pair with `services.service` | Pair with `services.service` |

The structured `service_unit` form currently supports these fields:

- `description`
- `run`
- `type`
- `user`
- `group`
- `working_dir`
- `environment`
- `restart`
- `restart_delay`
- `wants`
- `after`
- `wanted_by`
- `stdout`
- `stderr`

`wanted_by` defaults to `["multi-user.target"]`, allowing
`services.service.enabled = true` to enable the service directly. To generate a
unit without an install section, explicitly set `wanted_by = []`.

See `examples/systemd-service-unit.dbf.hcl` for a complete example.
