# `script` / `on_change` Requirements

<p align="right"><strong>English</strong> | <a href="script-on-change-requirements.zh.md">简体中文</a></p>

This document describes the design goals for the `script` directive and
`files.file.on_change`. DSL parsing, validation, HostSpec compilation,
ResourceGraph/plan operation presentation, apply-time script execution,
`mode = "once"` / `"each"` trigger semantics, and runtime trigger-context
injection are implemented. See the [DSL Reference](dsl-reference.md) for
supported syntax and the
[`script` / `on_change` implementation plan](script-on-change-implementation-plan.md)
for the staged implementation record.

## Core Concepts

DebianForm's execution model centers on the `host`:

- A `host` is the final execution unit. `plan`, `apply`, and `check` target
  hosts, each with its own SSH connection, remote state, and lock.
- A `profile` is a non-parameterized base-configuration fragment for reusable
  host configuration. A profile can be imported by another profile or a host.
  Imported content is merged first; the current profile or host is merged last
  and overrides fields with the same name.
- A `component` is a parameterized reusable deployment unit that encapsulates
  resources, exposes typed inputs, and may declare artifact download, build,
  and installation. A component expands only after a host mounts it. It neither
  has the complete semantics of a host nor executes independently.

`script` / `on_change` belongs to a component's internal runtime lifecycle: a
component declares how its own changes should take effect after its files
change. It does not belong to profiles and should not be a general host callback
mechanism.

This runtime lifecycle differs from the existing
`lifecycle { prevent_destroy = true }`: the latter protects resource deletion,
whereas the former is a post-change operation hook.

## Goals

- Let a component encapsulate configuration files and their corresponding
  reload, restart, or activation operation.
- Let callers customize behavior through inputs instead of passing an external
  script reference.
- Preserve clear host/profile/component boundaries: hosts compose, profiles
  reuse base configuration, and components own their internal activation logic.
- Make plans show clearly which file changes trigger which operations.
- Prevent script content from leaking sensitive file content or making plan
  output excessively long.

## Non-Goals

- The first version does not allow `script` declarations in a `host` or
  `profile`.
- The first version does not let a host pass a `script` reference as a component
  input.
- The first version does not let one `files.file` bind multiple `on_change`
  callbacks.
- The first version does not make `script` an independently applicable
  top-level resource.

## DSL Design

A `script` appears inside a `component` at the same level as `files`,
`services`, and similar blocks.

```hcl
component "app" {
  input "service_name" {
    type = string
  }

  script "reload" {
    mode = "once"
    run  = "systemctl reload ${input.service_name}.service"
  }

  files {
    file "/etc/app/config.yaml" {
      content   = "..."
      on_change = script.reload
    }
  }
}
```

`files.file.on_change` accepts exactly one script reference. One final file
change triggers one on-change operation. The script itself coordinates any
multi-step behavior.

## Script Fields

`script "<name>"` supports:

| Field | Required | Default | Description |
| --- | --- | --- | --- |
| `mode` | No | `"once"` | `"once"` or `"each"`. |
| `interpreter` | No | `["/bin/sh", "-eu"]` | Interpreter and arguments used to execute script content. |
| `outputs` | No | `[]` | Absolute paths to ordinary files generated by the script, used for check/plan drift detection. |
| `run` | Exactly one of three | None | A single shell-command string. |
| `content` | Exactly one of three | None | Multiline script text. |
| `commands` | Exactly one of three | None | A command matrix such as `[["systemctl", "reload", "app.service"]]`. |

`run`, `content`, and `commands` are mutually exclusive, and exactly one is
required.

Intended semantics:

- `run` is wrapped and executed as script text.
- `content` is passed unchanged to `interpreter`.
- DebianForm safely assembles `commands` as script text, with each inner list
  representing a command and its arguments.

## Execution Modes

`mode = "once"`:

- If several files trigger the same script during one apply, it runs once.
- Suitable for coalescing operations such as `systemctl reload app.service` or
  `systemctl restart app.service`.

`mode = "each"`:

- The script runs once for every triggering file during one apply.
- Suitable when the script processes each file path individually, for example
  validating or importing each generated file.

A script failure fails apply. This follows the existing strict operation model
and provides no warn-only behavior.

## Output Files

`outputs` brings script side effects into the DebianForm state model. Every
output must be an absolute path and must exist as an ordinary file after the
script succeeds. DebianForm records its SHA-256. A later check or plan retriggers
the script when an output is missing, becomes a directory, differs from the
recorded hash, or when the script declaration itself changes.

`outputs` does not mean DebianForm writes those files directly, and removing the
declaration does not delete remote output files. It only defines the boundary
for checking and reconverging script output.

## Runtime Context

When running a script, DebianForm injects environment variables so scripts do
not need to hard-code their trigger source:

| Environment variable | Description |
| --- | --- |
| `DBF_SCRIPT_NAME` | Script name. |
| `DBF_COMPONENT_NAME` | Component instance name. |
| `DBF_TRIGGER_ADDRESS` | Current triggering resource address; always one address in `each` mode. |
| `DBF_TRIGGER_PATH` | Current triggering file path; always one path in `each` mode. |
| `DBF_TRIGGER_ADDRESSES` | Newline-separated triggering addresses in `once` mode. |
| `DBF_TRIGGER_PATHS` | Newline-separated triggering paths in `once` mode. |

Component inputs already participate in expression evaluation while the
component is compiled, so ordinary customization should use inputs.

## Complete Example

This example lets a component encapsulate its configuration file, systemd unit,
service, and reload behavior. The host only mounts the component and supplies
ordinary inputs.

```hcl
component "managed_app" {
  input "service_name" {
    type = string
  }

  input "listen_addr" {
    type    = string
    default = "127.0.0.1:8080"
  }

  script "reload" {
    mode = "once"
    run  = "systemctl reload ${input.service_name}.service"
  }

  files {
    file "/etc/managed-app/config.env" {
      owner = "root"
      group = "root"
      mode  = "0644"

      content = "LISTEN_ADDR=${input.listen_addr}\n"

      on_change = script.reload
    }
  }

  systemd {
    service_unit "managed-app" {
      description = "Managed app"
      run         = ["/usr/local/bin/managed-app", "--config", "/etc/managed-app/config.env"]
      restart     = "always"
    }
  }

  services {
    service "managed-app" {
      enabled = true
      state   = "running"
    }
  }
}

host "app1" {
  component "app" {
    source = component.managed_app

    inputs = {
      service_name = "managed-app"
      listen_addr  = "127.0.0.1:9000"
    }
  }
}
```

When different environments should restart rather than reload, model that
behavior as an ordinary input:

```hcl
component "managed_app" {
  input "service_action" {
    type    = string
    default = "reload"

    validation {
      condition     = contains(["reload", "restart"], input.service_action)
      error_message = "service_action must be reload or restart."
    }
  }

  input "service_name" {
    type = string
  }

  script "apply_config" {
    mode = "once"
    run  = "systemctl ${input.service_action} ${input.service_name}.service"
  }

  files {
    file "/etc/managed-app/config.env" {
      content   = "..."
      on_change = script.apply_config
    }
  }
}
```

## Compilation and Execution Model

The intended compilation pipeline is:

1. Merge hosts and profiles using the existing rules to produce base host
   configuration.
2. Mount components on the host and evaluate inputs for each component instance.
3. Resolve component-local `script` and `files.file.on_change` references in
   the instance scope.
4. Build dependency edges between file nodes and script operations in the
   ResourceGraph.
5. During online planning, only actual file creates, updates, and deletes
   trigger their operations.
6. During apply, apply file changes first and run triggered script operations
   in dependency order.

Operation addresses should include the component instance so repeated mounts of
the same component cannot collide:

```text
host.app1.components.app.script["reload"]
```

## Plan Presentation

Plans currently show short summaries without expanding full script content:

```text
  ! host.app1.components.app.script["reload"]
    run component script reload
    triggered_by: host.app1.components.app.files.file["/etc/managed-app/config.env"]
    command: script reload (once)
```

The current ResourceGraph and plan expose only a short `command_preview`. The
full execution payload stays in an internal operation field in memory for apply
and is not part of the plan text/JSON/HTML public interface.

## Open Questions

- Should `commands` remain a first-version capability, or should the first
  version support only `run` and `content`?
- Should similar operations be coalesced across components, such as one nginx
  reload triggered by several components?
- For multiple triggers in `mode = "once"`, are environment-variable lists
  sufficient, or should JSON on stdin be a later extension?
